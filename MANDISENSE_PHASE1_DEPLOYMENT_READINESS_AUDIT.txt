SECTION 1 â€” REPOSITORY HEALTH AUDIT
================================================================================

1. Folder Structure & File Organization:
   - Root Directory Layout: Contains both modular project folders (`api`, `backend`, `frontend`, `mandisense_ai`) and a clutter of presentation files, build artifacts, local scripts, and execution logs.
   - Nested Architecture: The repository contains a sub-application structure inside `mandisense_ai/` which possesses its own duplicate configuration files (`Dockerfile`, `docker-compose.yml`, `nginx.conf`, `.gitignore`, `pyproject.toml`, `.env.example`, `main.py`). This causes major configuration drift.

2. Naming Consistency:
   - Mixed conventions are present across folders and files. Core Python services use snake_case (`mandisense_ai`), while system configurations use kebab-case (`pyproject.toml`, `docker-compose.yml`, `render.yaml`).

3. Deployment Artifacts:
   - Root `Dockerfile`: Authoritative for backend uvicorn deployment but lacks separation of packaging layers (excludes `backend` from package discovery in `pyproject.toml`).
   - Root `docker-compose.yml`: Launches DB, Redis, and API. Conflicts with `mandisense_ai/docker-compose.yml` which defines extra services (Prometheus, Grafana, Nginx).
   - Root `render.yaml`: Provisions free-tier database, redis, and uvicorn container. Lacks a database schema initialization migration step.

4. Legacy and Dead Files:
   - `mandisense_ai/api/`: Completely dead code. Contains legacy router definitions, lifespan app factory, and configuration handlers.
   - `mandisense_ai/main.py`: Outdated script that duplicates uvicorn launching.
   - `backend/app/main.py`: Legacy standalone entry point.
   - `generate_elite_presentation_script.py` & `generate_presentation_script.py`: Local scratch scripts.
   - `slide_parse.txt` & `slide_parse_temp.py`: Local slide parser files.
   - `mandisense_ai_elite_presentation_script.docx` & `mandisense_ai_presentation_script.docx`: Large binary documents.
   - `test_payload.json` & `mandi_master_temp.csv`: Local test data files.
   - Packaging metadata: Tracked `mandisense_api.egg-info/` and `mandisense_ai/mandisense_ai.egg-info/` folders.

5. Duplicate Files:
   - Configuration: Root `Dockerfile` vs `mandisense_ai/Dockerfile`.
   - Orchestration: Root `docker-compose.yml` vs `mandisense_ai/docker-compose.yml`.
   - Git ignores: Root `.gitignore` vs `mandisense_ai/.gitignore` vs `frontend/.gitignore`.
   - Settings: `backend/config/settings.py` vs `mandisense_ai/config/settings.py`.

6. Audit Recommendations:
   - What should remain: `api/` (root), `backend/` (root, minus app/main.py), `frontend/`, `mandisense_ai/` (minus legacy api/ and config files), root `Dockerfile`, root `docker-compose.yml`, root `render.yaml`, root `.gitignore`, root `.dockerignore`.

   - What should be archived: `presentation/`, `presentation_assets/`, `models_backup_20260503_1905/`, and `scratch/` should be archived outside the active production deployment repository.

================================================================================

1. Finding 1: News API Key
   - Location: `mandisense_ai/core/agents/external_factors_agent/orchestration/config_manager.py` (Line 5)
   - Severity: CRITICAL
   - Risk: Hardcoded API key `0cf53d3d3f554be09c9b20505aeeefb9` is exposed. When made public, the key will be immediately scraped, exhausted, or revoked.
   - Recommended Fix: Load the API key from environment variables:
     self.api_keys = {"news_api": os.getenv("NEWS_API_KEY", "")}

2. Finding 2: Fallback Database URL
   - Location: `mandisense_ai/db/connection.py` (Line 33)
   - Severity: WARNING
   - Risk: Hardcoded fallback connection string `"postgresql://user:pass@localhost:5432/mandisense"` is defined. While localhost defaults are not exploits, standard security compliance requires throwing an error if the URL is not supplied in a production environment.
   - Recommended Fix: Fallback to empty string and fail fast if DATABASE_URL environment variable is missing when running in production.

3. Finding 3: Fallback Redis URL
   - Location: `backend/config/settings.py` (Lines 5-8)
   - Severity: SAFE
   - Risk: Default localhost connection parameters. Non-exploitable locally, overridden in production.

No other database passwords, private keys, JWT secrets, or AWS keys were found.

     - `LOGGING__LEVEL`: Optional. Set to `INFO` in production.
     - `CORS_ALLOW_ALL_ORIGINS`: Optional. Set to `true` to allow frontend web access, or `false` with explicit trusted domains.
     - `RATE_LIMIT_PER_SECOND`: Optional. Set API request limits (default `60`).
   - Frontend (Vercel Host):
     - `NEXT_PUBLIC_API_URL`: Required. The domain of the deployed FastAPI backend (e.g. `https://mandisense-api.onrender.com`).

SECTION 5 â€” FRONTEND DEPLOYMENT READINESS
================================================================================

Output: NOT DEPLOYABLE

Explanation:
1. Hardcoded API Base Path: In `frontend/src/services/api.ts` and `frontend/src/hooks/useCognitionStream.ts`, the application defaults `API_BASE_URL` to `http://localhost:8000`. Without setting the `NEXT_PUBLIC_API_URL` environment variable during deployment, client-side requests and WebSocket connections will attempt to hit localhost, crashing the client interface.
2. Invalid Next.js Dependency: The `package.json` specifies `"next": "16.2.4"`. Next.js 16 is a future, non-existent stable version (Next.js 15 is current). This invalid package configuration will break npm install on Vercel unless legacy overrides are utilized.
3. Tracked Local Files: TypeScript build cache `tsconfig.tsbuildinfo` is committed in Git, causing build state conflicts.
4. Missing Config Templates: There is no `.env.example` in the `frontend` folder to document variables.

SECTION 6 â€” BACKEND DEPLOYMENT READINESS
================================================================================

Output: NOT DEPLOYABLE

Explanation:
1. No Automated Database Migration: The database schema is defined only in `mandisense_ai/db/schema.sql`. There is no script that automatically runs this schema against PostgreSQL on startup, nor is there a Render migration command set up. A deployment to Render will result in an empty database, causing all operations to crash.
2. Hardcoded Key: The agent relies on a hardcoded API key in python scripts, which risks immediate revocation.
3. Package Packaging Gaps: The root `pyproject.toml` packages list includes only `api*` and `mandisense_ai*`, but the active server `api/main.py` imports modules from the `backend/` directory. If setuptools packages the app, the `backend` directory is excluded, which will result in `ModuleNotFoundError` at runtime unless the container preserves the raw folder structure and leverages `PYTHONPATH`.
4. Duplicate Entry Points: Multiple active servers exist (`api/main.py`, `backend/app/main.py`, `mandisense_ai/main.py`) with conflicting worker allocations (1 worker vs 2 workers), making it difficult for orchestrators to resolve which file is authoritative.

SECTION 9 â€” PRODUCTION BLOCKERS
================================================================================

1. CRITICAL: Hardcoded News API Key
   - Impact: Key compromise, usage quota exhaustion, and code security violation.
   - Fix: Load News API key via `os.getenv("NEWS_API_KEY")`.
   - Time: 5 minutes.

2. CRITICAL: Lack of Database Schema Initialization
   - Impact: App crash. Database will have no tables, and all query executions will fail.
   - Fix: Write an initialization script `scripts/init_db.py` to run the `schema.sql` queries on startup or use a Render pre-deploy command.
   - Time: 15 minutes.

3. HIGH: Missing Frontend Production Endpoint Config
   - Impact: Client fails to connect to the backend, rendering the app unusable.
   - Fix: Set `NEXT_PUBLIC_API_URL` on Vercel and create `frontend/.env.example`.
   - Time: 5 minutes.

4. HIGH: Tracked Cache, Models, and Large Snaps in Git
   - Impact: Repository bloat, git history pollution, and potential data leakage.
   - Fix: Run `git rm --cached` on caches, model picklers, and snapshots, and update `.gitignore`.
   - Time: 10 minutes.

5. MEDIUM: Non-existent Next.js Version in Package JSON
   - Impact: Deployment failure on Vercel due to npm registry package resolution failure.
   - Fix: Downgrade Next.js to stable `"next": "^15.1.0"` and `"react": "^19.0.0"`.
   - Time: 5 minutes.

6. MEDIUM: Packaging Configuration Omits Backend Directory
   - Impact: Potential import crashes when run as an installed package.
   - Fix: Add `"backend*"` to root `pyproject.toml` find packages option.
   - Time: 5 minutes.
