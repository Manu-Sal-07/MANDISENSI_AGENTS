2. SCHEMA STRATEGY
================================================================================

- The initialization strategy is completely idempotent and safe for continuous delivery.
- Uses `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` commands defined in `schema.sql` to avoid mutating or dropping pre-existing production tables.
- Verification queries catalog views in `information_schema.tables` to validate existence rather than relying on side-effect testing (e.g. SELECT 1).

================================================================================
4. INITIALIZATION LOGIC
================================================================================

- Container Level: `Dockerfile` executes `python scripts/init_db.py` on container boot. This uses the `psycopg2` sync connector to connect, execute `schema.sql`, and assert table existence.
- Backend Level: `api/main.py` executes `verify_and_initialize_db` in the `@app.on_event("startup")` hook before uvicorn begins accepting traffic. This uses `asyncpg` to verify the schema.
- Fail-Fast Mechanism: If verification fails, in a `production` environment, the process logs a critical error and calls `sys.exit(1)`. This immediately terminates the container, prompting the cloud platform (Render) to abort deployment and roll back.

================================================================================
5. RENDER CHANGES
================================================================================

- Release command integration is handled natively inside the Docker container command, eliminating the need to set up a separate and slower external release process in `render.yaml`.
- Health check requests hitting `/v1/health` return HTTP 503 Service Unavailable if any core subsystem circuit breaker (Postgres or Redis) transitions to the `"OPEN"` state.
